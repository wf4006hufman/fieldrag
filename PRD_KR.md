# PRD — FieldRAG: NGS 필드 지원용 근거 기반 RAG + 에이전트 어시스턴트

**소유자:** 정웅재
**이 문서의 목적:** FieldRAG가 *무엇이고* 포트폴리오 프로젝트로서 *왜* 존재하는지 정의해서, 범위를 집중시키고 결과물이 일관되고 자기 완결적인 실증(demonstration)이 되게 한다.

---

## 1. 이 프로젝트가 존재하는 이유

1. **응용 GenAI 엔지니어링의 엔드투엔드 실증.** FieldRAG는 완결된 근거 기반 RAG 시스템이 Google Cloud 위에서 처음부터 끝까지 동작함을 보여준다: 문서 인제스트 + 임베딩 파이프라인을 벡터 DB로, 시맨틱 검색, 출처 인용이 붙은 근거 기반 생성, 다단계 에이전트, 평가 하네스, 관측성 — 그리고 이 모두를 GCP에 라이브로 배포. 장난감 챗봇이 아니라, 각 조각이 실제로 맞물려 돌아간다는 증거로서 그 자체로 성립하도록 만든 것이다.

2. **범용이 아니라 도메인에 맞게.** 범용 어시스턴트 대신, FieldRAG는 **공개 NGS/생명정보 문서**(변이 콜링 가이드, 파이프라인 문서, 툴 매뉴얼)에 근거해 답한다. 데모를 실제 기술 도메인에 근거시키면 검색 품질과 인용 동작이 의미를 갖게 되고, 지원/필드 팀이 실제로 만들 법한 내부 지식 어시스턴트를 반영한다.

## 2. 무엇인가 (한 문장)

필드 엔지니어가 자연어로 질문하면 Gemini가 큐레이션된 기술 문서 코퍼스에서 **근거 기반 + 출처 인용 답변**을 합성해 주는 웹/API 서비스. 검색은 벡터 데이터베이스 위 시맨틱 서치로 하고, 여러 단계를 밟으며 도구를 쓰는 **에이전트 모드**와, 실제로 동작함을 입증하는 **평가 + 관측성 계층**을 갖춘다.

## 3. 사용자 & 핵심 사용자 스토리

| 사용자 | 스토리 |
|---|---|
| 필드 엔지니어 (주) | "*'germline NGS 런에서 시료 오염을 나타내는 QC 지표는?'* 라고 물으면 **출처 문서 + 섹션이 인용된** 간결한 답을 받아 신뢰하고 전달할 수 있다." |
| 필드 엔지니어 | "두 번 조회가 필요한 질문(예: 두 도구 비교)에서는 내가 다시 묻지 않아도 **에이전트**가 다단계 검색을 해준다." |
| 유지보수자 | "`eval.py`를 돌리면 **검색 hit-rate와 groundedness 점수**가 보여서, 그럴듯한 게 아니라 정확한지 안다." |
| 운영자 | "Cloud Logging을 열면 **요청별 지연시간, 검색된 문서 ID, 토큰 사용량**이 보인다 — 진짜 관측성." |

## 4. 범위(Scope)

### 포함
- **인제스트 파이프라인**: 문서 로드 → 청킹 → 임베딩 → pgvector 저장.
- **RAG 쿼리**: 질문 임베딩 → 벡터 검색 top-k → Gemini 근거 기반 답변 + 인용.
- **에이전트 모드**: `search_docs` + `list_sources` 도구를 쓰는 LangGraph ReAct 에이전트.
- **평가**: golden Q/A 세트 → 검색 hit@k + LLM-judge groundedness 점수 → 리포트 출력.
- **관측성**: 구조화 JSON 로그(지연시간, 검색 ID, 토큰)를 Cloud Logging에 노출.
- **데모 UI**: 영어 문구를 사용하는 단일 FastAPI HTML 화면. RAG/Agent 모드 선택, 출처 표시, 검색 진단 정보 포함.
- **배포**: FastAPI를 컨테이너로 **Cloud Run**에, 벡터 저장소는 **Cloud SQL for PostgreSQL + pgvector**.
- **선택**: `search_docs`를 **MCP 서버**로 노출.

### 제외 (범위 가드레일)
- 인증 / 멀티테넌트 / 사용자 계정.
- 완전한 제품형 프론트엔드, 디자인 시스템, 대시보드 묶음, React/Vite 전환. UI는 FastAPI가 제공하는 단일 polished demo page로 유지한다.
- 파인튜닝, 리랭킹 모델, 하이브리드 검색, 스트리밍 응답.
- 독점/벤더 내부 문서 인제스트 — **공개 출처만** (IP 문제 회피, GitHub 공유 가능 유지).
- 비용 최적화, 오토스케일링 튜닝, CI/CD.

## 5. 성공 기준 (완료의 정의)

**필수:**
- [ ] `ingest.py`가 공개 문서를 pgvector에 로드; 행 수 검증.
- [ ] 질문 시 **최소 1개의 올바른 인용**이 붙은 근거 기반 답변 반환.
- [ ] 서비스가 **공개 Cloud Run URL**에서 Cloud SQL pgvector와 통신하며 라이브.
- [ ] `eval.py`가 golden 질문 ≥10개에 대해 검색 hit@k와 groundedness 점수 출력.
- [ ] Cloud Logging에 지연시간 + 검색 문서 ID가 담긴 요청별 구조화 로그 표시.
- [ ] 루트 웹 페이지가 답변, 출처, 모드, 지연시간, 검색 ID를 보여주는 포트폴리오용 데모 화면을 제공.

**있으면 좋음:**
- [ ] LangGraph 에이전트가 2-hop 질문에 정확히 답변.
- [ ] MCP 서버가 `search_docs` 노출.
- [ ] 아키텍처 다이어그램과 평가 수치가 담긴 짧은 `README.md`.

## 6. 완성된 프로젝트가 보여주는 것

정직하고 검증 가능하게 실증하는 하나의 자기 완결적 산출물:

> **FieldRAG** — Gemini 기반 RAG 어시스턴트를 **GCP Cloud Run**에 **pgvector** 벡터 DB 위로 배포, **LangGraph** 에이전트, **LLM-judge 평가 하네스**(검색 hit-rate + groundedness), **Cloud Logging** 관측성 포함.

이 설명의 모든 구성 요소는 이 저장소에 구현되어 있고, README에 링크된 라이브 배포에서 실제로 동작한다.

## 7. 리스크 & 완화

| 리스크 | 완화 |
|---|---|
| Cloud SQL 네트워킹 복잡도 | Cloud Run 내장 `--add-cloudsql-instances`(Unix 소켓) 사용 — VPC/프록시 설정 불필요. 대안: pgvector를 로컬 Docker Postgres에 두고 앱만 Cloud Run에 배포. |
| GCP 요금 | 데모에는 `gemini-2.5-pro`를 Gemini Developer API / visitor BYOK 경로로 사용해 free-tier를 적용하고, Cloud SQL은 최소 티어로 두며, 끝나면 **리소스 삭제**(가이드의 teardown). |
| 모델 ID 변경 | 가이드는 모델 ID를 `app/config.py`에 모아둠; 호출이 404면 Model Garden에서 확인. |
| 범위 확장(scope creep) | 로컬에서 동작하는 RAG 경로가 먼저다. 나머지는 그게 초록불이 된 뒤에만 얹는다. |
